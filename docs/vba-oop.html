Fondamentaux

Pourquoi la programmation orientée objet ?

La différence entre du code procédural (VBA classique) et du code orienté objet — avec le même exemple.

❌ VBA Procédural — le chaos

Tout est à plat. N'importe qui modifie n'importe quoi.


Sub PricerOption()
  Dim underlying As String
  Dim strike As Double
  Dim vol As Double
  Dim spot As Double
  Dim prix As Double
  Dim delta As Double

  underlying = "TTE.PA"
  strike = 60.0
  vol = 0.28
  spot = 58.40
  ' 50 variables éparpillées...
  ' Qui gère la validation ?
  ' Que se passe-t-il si strike = -5 ?
  ' Rien ne l'empêche.
  strike = -5  ' BUG silencieux
End Sub

✅ VBA OOP — la maîtrise

Chaque classe protège ses données. La validation est intégrée.


Sub PricerOption()
  Dim opt As New COption

  opt.Underlying = "TTE.PA"
  opt.Strike = 60.0   ' validé : doit être > 0

  opt.Strike = -5
  ' → Erreur 102 levée automatiquement
  ' → "Strike doit être > 0"
  ' → Impossible de corrompre les données

  ' L'objet se protège lui-même.
End Sub
Anatomie

Anatomie d'une classe VBA — COption disséquée

Chaque bloc de code a un rôle précis. Voici exactement ce qui se passe en mémoire.

1. Les champs privés — la mémoire interne

Ce sont les variables membres de l'objet. Déclarées Private → personne ne peut y accéder directement depuis l'extérieur. La convention m_ signale que c'est un membre (member). 


' === CHAMPS PRIVÉS ===
Private m_underlying   As String   ' "TTE.PA" — inaccessible directement
Private m_optionType   As String   ' "CALL" ou "PUT"
Private m_strike       As Double   ' 60.00 — on ne peut pas écrire opt.m_strike

' Ce que ça interdit :
' opt.m_strike = -5   → Erreur de compilation, m_strike est Private
Principe : Encapsulation

Les données sont cachées à l'intérieur de la classe. On ne peut les lire/modifier QUE via les propriétés publiques — qui font la validation.

2. Class_Initialize — le constructeur

S'exécute automatiquement quand tu fais New COption. C'est le constructeur VBA — il initialise les valeurs par défaut. 


Private Sub Class_Initialize()
    m_state   = "SAISI"    ' état initial de l'option
    m_style   = "EU"       ' européenne par défaut
    m_buySell = "BUY"      ' achat par défaut
End Sub

' Ce qui se passe quand tu fais :
Dim opt As New COption
'         ↑
'         Class_Initialize() est appelé automatiquement
'         opt.State   = "SAISI"  sans que tu aies rien fait
'         opt.Style   = "EU"
'         opt.BuySell = "BUY"

3. Property Get / Let — les gardiens

Ce sont les accesseurs. Get = lire. Let = écrire avec validation. C'est ici que la classe protège ses données. 


' LIRE (Get) — pas de validation nécessaire
Public Property Get Strike() As Double
    Strike = m_strike          ' retourne la valeur interne
End Property

' ÉCRIRE (Let) — validation obligatoire
Public Property Let Strike(v As Double)
    If v <= 0 Then
        Err.Raise vbObjectError + 102, "COption", "Strike doit être > 0"
        '         ↑ numéro unique    ↑ source     ↑ message clair
    End If
    m_strike = v               ' stockage seulement si valide
End Property

' Utilisation :
opt.Strike = 60.0    ' → Property Let Strike(60.0) → OK, m_strike = 60
opt.Strike = -5      ' → Property Let Strike(-5)   → Err.Raise → exception
Dim k = opt.Strike   ' → Property Get Strike()     → retourne 60.0

4. Propriété calculée — Get sans Let

Pas de valeur stockée — calculée à la volée. Lecture seule : pas de Let. 


Public Property Get Maturity_Years() As Double
    ' Calculé dynamiquement à chaque appel
    ' Pas de m_maturity_years stocké en mémoire
    Maturity_Years = (m_maturity - Date) / 365#
End Property

' Comportement :
' opt.Maturity = 09/06/2026, aujourd'hui = 10/03/2026
' opt.Maturity_Years → (09/06/2026 - 10/03/2026) / 365 = 0.252

' Demain :
' opt.Maturity_Years → (09/06/2026 - 11/03/2026) / 365 = 0.249
' La valeur change automatiquement avec le temps !
Architecture

La hiérarchie de classes — comment elles se parlent

CApp est le chef d'orchestre. Elle crée et expose les sous-systèmes. C'est le pattern Composition.

La RAM quand CApp est instanciée


Dim app As New CApp
' ↑ Cette seule ligne crée toute cette structure en RAM :

RAM
│
└── app (CApp)                        ← objet racine
    ├── m_config (CConfig)            ← sous-objet créé par CApp
    │   └── m_params (Dictionary)
    │       ├── "DEFAULT_RISK_FREE" → 0.035
    │       ├── "DEFAULT_VOL"       → 0.25
    │       ├── "LOT_SIZE"          → 100
    │       └── "CURRENCY"          → "EUR"
    │
    ├── m_marketData (CMarketData)    ← sous-objet créé par CApp
    │   └── m_spots (Dictionary)
    │       ├── "TTE.PA" → 58.40
    │       ├── "BNP.PA" → 72.15
    │       └── ... (6 tickers)
    │
    ├── m_pricingEngine (CPricingEngine) ← sous-objet créé par CApp
    │   └── (pas de données, que des méthodes)
    │
    ├── m_database (CDatabase)        ← sous-objet créé par CApp
    │   └── m_conn (ADODB.Connection) → connexion Access ouverte
    │
    └── m_blotter (CBlotter)          ← sous-objet créé par CApp
        └── m_ws (Worksheet)          → référence vers feuille Excel

L'appel hiérarchique — chaîne complète

Quand Sophie clique sur PRICER, voici exactement ce qui s'exécute, couche par couche : 


' BtnPricer appelle CApp
app.MarketData.GetSpot("TTE.PA")
'   ↑           ↑        ↑
'   CApp        Property  méthode de CMarketData
'               Get qui   qui cherche dans le
'               retourne  Dictionary m_spots
'               m_marketData

' Décomposé :
' 1. app               → objet CApp en RAM
' 2. .MarketData       → Property Get MarketData() → retourne m_marketData
' 3. .GetSpot("TTE.PA")→ méthode de CMarketData → cherche m_spots("TTE.PA") → 58.40

' Même principe :
app.Config.GetParam("DEFAULT_RISK_FREE")
' 1. app               → CApp
' 2. .Config           → retourne m_config (CConfig)
' 3. .GetParam(...)    → cherche m_params("DEFAULT_RISK_FREE") → 0.035

app.PricingEngine.Price(58.4, 60, 0.035, 0.28, 0.25, "CALL")
' 1. app               → CApp
' 2. .PricingEngine    → retourne m_pricingEngine (CPricingEngine)
' 3. .Price(...)       → calcule Black-Scholes → 2.4436

Cycle de vie d'un objet — naissance et mort

En VBA, les objets ont un cycle de vie géré automatiquement. 


Sub BtnPricer()
    Dim app As New CApp    ' ← NAISSANCE : Class_Initialize() appelé
    '                            CConfig, CMarketData, etc. créés en RAM
    '                            Connexion Access ouverte

    ' ... tout le code de pricing ...

End Sub  ' ← MORT : quand app sort de portée
         '    Class_Terminate() appelé automatiquement
         '    Set m_config = Nothing → CConfig détruite
         '    Set m_database = Nothing → connexion Access FERMÉE
         '    Toute la RAM libérée proprement

' VBA gère ça tout seul → pas de fuite mémoire si on utilise Class_Terminate
Principe : RAII (Resource Acquisition Is Initialization)

La connexion Access est ouverte à la création de CDatabase et fermée à sa destruction. Automatique. Propre. Zéro connexion orpheline.

Principes SOLID

Single Responsibility — chaque classe fait une seule chose

📦

COption

Fait : représenter une option avec ses paramètres contractuels + validation.

Ne fait pas : pricer, accéder à la base, afficher dans Excel.

⚙️

CPricingEngine

Fait : calculer le prix Black-Scholes et les Greeks à partir de 5 paramètres.

Ne fait pas : lire les données de marché, stocker quoi que ce soit.

🗃️

CDatabase

Fait : sauvegarder et lire des données dans Access via ADODB.

Ne fait pas : pricer, afficher dans Excel, valider des options.

📈

CMarketData

Fait : fournir les prix de marché (spots, taux) — simulés ou connectés Bloomberg.

Ne fait pas : pricer, calculer des Greeks.

📊

CBlotter

Fait : afficher les positions dans la feuille Excel Blotter.

Ne fait pas : aller chercher les données elle-même — elle reçoit un CDatabase.

🎯

CApp

Fait : créer, initialiser et exposer tous les sous-systèmes. Chef d'orchestre.

Ne fait pas : aucune logique métier — elle délègue tout.

Robustesse

Gestion des erreurs OOP — Err.Raise vs On Error

Dans la classe — lever une erreur


' Dans COption.cls
Public Property Let Strike(v As Double)
    If v <= 0 Then
        Err.Raise _
          vbObjectError + 102, _ ' numéro unique
          "COption.Strike",    _ ' source
          "Strike doit être > 0" ' message
    End If
    m_strike = v
End Property

' Règle : la classe lève l'erreur
' Elle ne l'attrape PAS — c'est à l'appelant de gérer

Dans l'appelant — attraper l'erreur


' Dans BtnPricer ou TestModule
Sub BtnPricer()
    On Error GoTo ErrHandler   ' filet de sécurité

    Dim opt As New COption
    opt.Strike = CDbl(ws.Range("D6").Value)
    ' Si la cellule contient -5 → COption lève Err 102
    ' → On Error GoTo ErrHandler l'attrape ici

    ' ... reste du code ...
    Exit Sub

ErrHandler:
    MsgBox "Erreur : " & Err.Description, vbCritical
    ' Affiche "Strike doit être > 0"
End Sub
