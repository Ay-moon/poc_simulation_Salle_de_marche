Architecture en 4 couches multi-langages

Principe : chaque technologie fait ce qu'elle fait le mieux. VBA pour la proximité Excel/trader, .NET pour les calculs intensifs, Python pour l'analytique, SQL pour la persistance.

💬
Pourquoi ce découpage multi-langages ?
Un quant dev en banque travaille rarement avec un seul langage. VBA reste incontournable pour Excel. C# et VB.net dominent le pricing .NET en salle de marché. Python est le standard pour l'analytique quantitative. SQL Server est l'unique source de données — robuste, transactionnel, multi-utilisateurs.

🎓 Junior, écoute bien : Imagine que chaque langage est un spécialiste. VBA, c'est le commercial qui parle à Excel (les traders adorent Excel). C#, c'est l'ingénieur qui fait les calculs compliqués (Black-Scholes, c'est des maths). Python, c'est le statisticien qui fait les simulations (la VaR). SQL Server, c'est l'archiviste qui garde tout en sécurité. Chacun fait ce qu'il fait le mieux. Voilà pourquoi on a 4 langages différents !

Interop

Interopérabilité VBA ↔ .NET ↔ Python

Le vrai challenge technique : faire parler les langages entre eux de façon fiable et performante.

🔌

VBA ↔ C# (COM Interop)

La DLL C# est enregistrée comme serveur COM. VBA y accède via CreateObject("CommandoQuant.PricingEngine"). La DLL expose des interfaces simples (primitives + arrays). Le marshaling COM gère la conversion des types.

[ComVisible(true)]
[ClassInterface(ClassInterfaceType.None)]
public class PricingEngine : IPricingEngine
🐍

VBA ↔ Python (xlwings / Shell)

Deux approches selon le contexte :
1. xlwings : bibliothèque Python qui expose les fonctions Python directement dans Excel/VBA via un UDF server.
2. Shell + JSON : VBA lance Python via Shell(), passe les paramètres en JSON, lit les résultats dans un fichier.

🗄️

C# / Python ↔ SQL Server (ADO.NET / pyodbc)

C# utilise System.Data.SqlClient (ADO.NET) pour SQL Server. Python utilise pyodbc avec "ODBC Driver 17". Les deux passent par des stored procedures pour les opérations complexes. Connection string : Server=PERSO-AJE-DELL\BFASERVER01;Database=CommandoQuant;Trusted_Connection=yes.

Junior : "ADO.NET" et "pyodbc" c'est juste le nom des bibliothèques qui permettent au code de parler à SQL Server. Comme un traducteur entre C# et la base de données.
💬
Flux de données dans le pipeline de pricing
La séquence détaille comment les données transitent entre chaque couche lors du calcul d'une option. Le focus est sur la fiabilité : chaque appel inter-langage retourne un code d'erreur, les données sont sérialisées en JSON ou passées via des arrays COM simples.

Design Patterns

Patterns de conception — Cross-langage

Les mêmes patterns classiques s'appliquent quelle que soit la technologie.

🎯

Strategy — Modèle de pricing

Chaque modèle de pricing (BS, Binomial, MC) est une stratégie. IPricingModel définit le contrat. PricingEngineFactory sélectionne la stratégie selon le type de produit. Ajouter un modèle Heston = ajouter une stratégie, sans toucher au reste.

IPricingModel.Price(option, md)
IPricingModel.Greeks(option, md)
📦

Repository — Accès SQL Server

Sépare la logique métier de la persistance. IPositionRepository en C# définit le contrat. CSqlPositionRepository est l'unique implémentation — sur SQL Server PERSO-AJE-DELL\BFASERVER01. Access décommissionné.

Junior : Le "Repository Pattern", c'est comme demander à un assistant d'aller chercher tes fichiers. Tu lui dis "donne-moi la position POS-042" — tu ne sais pas s'il va dans une armoire, un coffre, ou un serveur. Il gère ça tout seul. Ici, l'armoire c'est SQL Server.
🔗

Composition Root (CApp)

En VBA, CApp est le seul endroit où tout est câblé. Lance le PricingEngine COM, injecte la config, démarre le logger, charge le cache de positions. ThisWorkbook.App est accessible depuis tous les modules sans new/init dispersé.

🏭

Factory — Création d'options

COptionFactory.Create(type, params) instancie la bonne classe selon le type de produit (EU, AM, Barrière, Asiatique). L'appelant ne connaît pas la classe concrète — il passe les params bruts, la factory gère la construction.

💾

Cache — Greeks & Positions

CPositionCache maintient un Dictionary VBA (positionId → CPosition + Greeks). Les lectures du blotter et du dashboard Greeks utilisent le cache. Invalidé à chaque reprice ou modification de position. x10 en performance sur 500+ positions.

🔄

Observer — Alertes temps réel

CRiskService observe les Greeks agrégés. Dès qu'une limite est franchie, il notifie tous les abonnés (Risk Manager dashboard, logger, rapport d'exception). Découplage total entre le calcul et la notification.
