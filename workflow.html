State Machine

Les 6 états d'une position en dérivés actions

Chaque position passe par une séquence d'états contrôlée par CWorkflowEngine. Aucun saut d'état sans validation des règles métier.

SAISI → VALIDÉ → PRICÉ → HEDGÉ → ACTIF → BOUCLÉ
💬
Pattern "State Machine" dans CWorkflowEngine
Le workflow utilise le pattern State : chaque état connaît les transitions autorisées et les conditions. La méthode CWorkflowEngine.Transition(position, newState) vérifie les pré-conditions, exécute les actions (calcule les Greeks, crée le hedge, log), puis change l'état. Impossible de passer de SAISI directement à HEDGÉ.

Pipeline EOD

Pipeline de calcul quotidien — End Of Day

Chaque soir, le pipeline EOD recalcule toutes les positions, les Greeks, la VaR et génère les rapports de risque.

⏰
Séquence EOD — 17h30 Paris
À la clôture des marchés, le pipeline récupère les cours de clôture, reprice toutes les positions, calcule les Greeks agrégés par book, puis lance les calculs VaR et stress tests. Le tout se déroule en séquence automatisée, de Excel VBA (orchestration) vers C# (pricing) et Python (VaR).
timeline title Pipeline EOD — Calcul Risque Quotidien 17h30 : Clôture marché : Récupération cours de clôture + vol implicite close 17h45 : Reprice positions : CBlackScholesEngine.PriceAll() — toutes positions actives 18h00 : Greeks consolidés : CGreeks.AggregateBook() — Δ Γ ν θ par book/desk 18h15 : VaR quotidienne : Python VaRCalculator — 3 méthodes en parallèle 18h30 : Stress tests : Python StressTester — scénarios macro + historiques 18h45 : P&L attribution : Décomposition Δ+Γ+ν+θ vs P&L réel 19h00 : Rapports générés : Risk report + P&L report + VaR report exportés

Exceptions

Cas d'exception & alertes risque

Les cas non nominaux doivent être gérés explicitement. CRiskService surveille les limites et émet les alertes.

⚠️

Dépassement limite Greek

Le Delta ou le Gamma du book dépasse la limite autorisée. CRiskService.CheckGreekLimits() compare les Greeks agrégés aux limites définies dans tbl_RiskLimits. Alerte immédiate au Risk Manager + blocage des nouvelles positions.

📉

Dépassement VaR

La VaR calculée dépasse la limite de VaR du desk. Déclenchement d'une procédure d'escalade : notification automatique, gel des nouvelles positions, rapport d'exception à la Direction des Risques. Traçabilité complète dans tbl_AuditLog.

🌊

Données de marché manquantes

Surface de vol incomplète ou cours absent pour un sous-jacent. CMarketDataService.Validate() détecte les trous. Fallback : interpolation ou utilisation de la dernière valeur disponible avec flag "STALE". Alerte Middle Office.

🔄

Model Override

Le Risk Manager remplace le prix modèle par un prix de consensus ou un prix de clôture différent. CPosition.ModelOverride = True. L'override est tracé dans tbl_AuditLog avec justification et approbation obligatoire.

📊

Backtesting Breach

La perte réelle dépasse la VaR prévue (exception de backtesting). Selon Bâle III, plus de 4 exceptions sur 250j → passage en zone rouge → capital supplémentaire requis. CommandoQuant comptabilise et alerte automatiquement.

📅

Expiration imminente

Une option arrive à expiration dans les 5 jours ouvrés. CWorkflowEngine émet une alerte : décision à prendre (dénouement anticipé, exercice, ou laissez expirer). Le Theta s'accélère → P&L impact fort si position non gérée.

Audit

Audit Trail — Traçabilité complète

Chaque action sur une position est tracée : qui, quoi, quand, depuis quel état, vers quel état, avec quelles données de marché.

🔏
Exigences réglementaires MiFID II / EMIR
Les régulateurs (AMF, ESMA) exigent une traçabilité complète des transactions et calculs de risque. Dans CommandoQuant, chaque action est loggée dans tbl_AuditLog : timestamp, user, positionId, action, paramètres, résultats. Immuable : append-only, pas de modification possible.
